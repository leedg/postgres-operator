# ADR-0004: CRD lifecycle is owned by the operator manager

- Date: 2026-05-02
- Status: Accepted
- Authors: @phil

## Context

Helm's `crds/` directory has *install-only* behavior. It is applied once during `helm install` and ignored at `helm upgrade` and `helm uninstall` time. This severs CRD evolution (field additions, validation tightening, conversion webhook adoption) from the chart upgrade flow, creating operational debt where users must run a separate procedure (`kubectl apply -f crds/`) manually. This operator's CRDs evolve from v1alpha1 → v1beta1 → v1 over a 6-year timeline (P0~P7) (see ADR-0005), and the Helm `crds/` model cannot provide safe evolution. At the same time, OLM (Operator Lifecycle Manager) recommends a model where the operator *directly owns* its CRDs, so this decision also simultaneously secures OLM integration friendliness.

## Decision

The operator manager (cmd/main.go) directly owns the CRD lifecycle by performing a server-side apply (SSA) at startup. The Helm chart's `crds/` directory is abandoned.

Key parameters:

- **CRD source**: embed `config/crd/bases/*.yaml` (generated by controller-gen) into the operator image (`//go:embed config/crd/bases/*.yaml`).
- **Apply timing**: once, just before `manager.Start()` in `cmd/main.go`, after acquiring leader election.
- **Apply method**: server-side apply (`fieldManager: postgres-operator`, `force: true`). Fields added by other field managers are preserved.
- **Owner annotation**: `postgres-operator.io/managed-by: operator`. Supports CRD change tracking and debugging.
- **Version conflict handling**: when the operator image is *downgraded* (e.g., v0.5.0 → v0.4.0), *skip* CRD apply (preserve storage version). Provide a force-override option via `kubectl annotate` (`postgres-operator.io/allow-crd-downgrade: "true"`).
- **Helm chart changes**:
  - **Remove** the `charts/postgres-operator/crds/` directory.
  - In `templates/NOTES.txt`, state "the CRDs appear automatically within about 5 seconds after the operator pod boots — verify with `kubectl get crd | grep postgres-operator`".
  - On `helm uninstall`, CRDs *remain* (intentional). Protects data. Explicit cleanup is performed by the user via `kubectl delete crd -l app.kubernetes.io/managed-by=postgres-operator`.
- **Developer tooling**: `make sync-crds` is *kept*. Used to verify the controller-gen → `config/crd/bases` → `docs/api-reference` sync. If the CRDs the operator embeds disagree with the docs, the CI L2 gate blocks the PR.
- **OLM integration**: this decision is compatible with all OLM `installModes` (`OwnNamespace` / `SingleNamespace` / `AllNamespaces`). The `customresourcedefinitions.owned` entry of the CSV (ClusterServiceVersion) is auto-generated by operator-sdk.

## Consequences

Positive:

- CRDs evolve automatically during `helm upgrade` — zero manual user steps.
- CRD/CR data is preserved automatically on `helm uninstall` — reduces operational accident risk.
- Friendly for OLM marketplace (OperatorHub.io) listing — the standard pattern where the operator owns its CRDs.
- Conversion webhook deployment (post-P5) is bound in lockstep with the operator, blocking *partial deployment* accidents.
- When CRD validation is tightened, only upgrading the operator applies it immediately.

Negative:

- Right after `helm install`, `kubectl get crd` may be empty — the user must wait for the operator pod to boot (~5 seconds). Mitigated by the NOTES.txt note.
- The operator pod requires CRD-apply permissions (cluster-scoped: `apiextensions.k8s.io/customresourcedefinitions:create,update,patch`). RBAC scope expands slightly.
- After `helm uninstall`, CRDs remain, so on *reinstall* the existing CR instances reappear — this is intended behavior but may surprise users. Document it in the uninstall NOTES.

Trade-offs:

- Immediate experience for *Helm-only users* vs. safe evolution for *OLM/long-term operations* users. The latter wins (single maintainer running for 6 years).
- RBAC expansion ↔ CRD lifecycle automation. Transparency for RBAC can be secured by an explicit `ClusterRole`.

## Alternatives Considered

| Alternative | Reason for rejection |
|---|---|
| (a) Keep the Helm `crds/` directory (current chart model) | Cannot evolve CRDs at upgrade time. During the 6-year v1alpha1 → v1 transition, manual user steps would explode. |
| (b) Use a `crd-install` Helm hook | Install-time CRD apply is possible but the upgrade hook has weak ordering guarantees and is a Helm 4 deprecation candidate. |
| (c) OLM only (no Helm) | Excludes direct K8s users (vanilla cluster, on-prem, edge). The ArtifactHub Helm channel becomes unusable. |
| (d) `kubectl apply` from a `pre-install` Job in a helm-charts repo | Job permissions, sa separation, and failure handling are complex. Integrating into the operator itself is simpler. |
| (e) Distribute a separate static CRD bundle (`postgres-operator-crds-1.0.0.yaml`) | Users must download it separately from the chart. No lockstep guarantee. |
| (f) The operator only *creates* CRDs and never updates them | Evolution is severed. Head-on conflict with the ADR's intent. |
